# 모듈별 알고리즘 및 난이도별 탐색 공간 분석 (상세판)

> 논문 작성용 데이터셋 특징 분석 문서.
> 모든 수치는 `generation/*.py` 의 난이도 설정 코드를 직접 읽고 도출했으며, 근거 `file:line` 을 명시한다.
> "탐색 공간(search space)" = **정답이 될 수 있는 후보의 가짓수**(answer candidate space). 솔버 내부 backtracking 노드 수와 구분한다.

# Part 1. 유형 A — 탐색공간 증가형

정답 후보 집합이 유한하며, 그 카디널리티가 난이도와 함께 증가한다. "탐색 공간 = 난이도" 가 직접 성립.

> ⚠️ **예외 — 1.6 inequality 는 유형 A\*(비단조·혼합)**: 후보 공간(size!/라틴방진)은 유한하나 카디널리티가 난이도와 단조롭지 않다(easy 1D 큰밴드 16!≈2.1×10¹³ > medium/hard 2D Futoshiki ≤8.1×10⁸; 게다가 easy↔medium 에서 1D→2D 로 문제 구조가 바뀜). 편의상 본 Part 에 두되 분류는 A\* 이며, 실제 난이도축은 카디널리티가 아니라 구조 전환/backtrack 비율이다(상세는 1.6 ⑤⑥, §4.2). **KO 도 재작성되어 EN 과 완전 동일(직번역).** 또한 본 Part 에 있던 **kinship 은 num_choices 26 고정으로 유형 B 로 재분류**됐다(절은 편의상 1.7 에 그대로 두고 ⑤⑥ 에 B 로 명시 — §17 참조).

## 1.1 Number Baseball (숫자야구) — `number_baseball_en.py` / `_ko.py`

**① 문제 정의 / 정답 형식**
N자리 정수(모든 자리 고유)를 strike(자리·숫자 일치)/ball(숫자만 일치) 힌트로 추론. 정답 형식 = N자리 distinct-digit 수 문자열 (예 "12345", `:547`).

**② 생성 파이프라인**
1. 비밀 숫자 생성: 0–9에서 중복 없이 N개 선택 (`:112-115`).
2. 힌트 후보 풀 구축: 비밀의 순열로부터 저-strike/고-ball 힌트 생성, 난이도별 선호 필터 (`:189-228`).
3. hard 경로: ball-중심 체인 전략으로 단계별 candidate filtering (ball≥2, strike≤1 우선, `:355-433`).
4. easy/medium 경로: 그리디 힌트 선택 + 2-step lookahead 로 잔여후보 최소화 (`:268-353`).
5. 유일해 검증: 힌트 적용 후 후보 정확히 1개 (`MAX_SOLUTIONS=1`, `:507-540`).

**③ 정답 도출 · 유일해 검증 알고리즘**
Brute-force 순열 필터링. 초기 후보 = `permutations('0123456789', N)` 전체 (`:39`). 각 힌트 (guess,S,B) 마다 후보 중 (strikes,balls) 일치만 유지 (`:139-147`). 유일성은 `find_all_solutions(hints, max_count=2)` 반환수=1 (`:149-151`). 복잡도 O(|hints|·P(10,N)). EN/KO 동일.

**④ 난이도 설정** (`DIFFICULTY_CONFIGS` `:130-208`, EN/KO 완전 동일) ※ 2026-06 전면 개정

| 파라미터 | easy | medium | hard |
|----------|------|--------|------|
| num_digits N | **7** | **8** | **8** |
| min/max_hints | 4/8 | 4/9 | 4/9 |
| **revealed_digits** (숫자집합 일부 공개) | **5** | 0 | 0 |
| **revealed_positions** (위치 고정, 기댓값 float) | – | **0.6** | – |
| preferred_strikes | (0,2) | (0,2) | (0,2) |
| preferred_balls | (0,5) | (0,6) | (0,6) |
| pool_size | 50 | 50 | 50 |

※ 구버전의 `target_residual`·`min_ball_heavy_ratio` 키는 **제거**됨. `revealed_positions` 는 float(기대 공개 위치 수) — medium=0.6 이면 60% 퍼즐만 1칸(위치→숫자) 공개 (`:670-681`).

**⑤ 탐색공간 분석**
정답 = N자리 distinct-digit 수 → **P(10,N) = 10!/(10−N)!** (`_get_candidate_space` `:36-41`)

| 난이도 | N | 공식 | nominal 가짓수 | reveal 후 effective |
|--------|---|------|----------------|---------------------|
| easy | 7 | P(10,7) | **604,800** | 숫자집합 5개 공개 → 대폭 축소 |
| medium | 8 | P(10,8) | **1,814,400** | 60% 퍼즐 1위치 공개 |
| hard | 8 | P(10,8) | **1,814,400** | 공개 없음(full) |

→ **medium 과 hard 는 nominal 공간이 1,814,400 으로 동일**하다(둘 다 8자리). 두 티어의 난이도 차이는 탐색공간이 아니라 **reveal**(medium 0.6 위치공개 vs hard 무공개)에서 온다 — 즉 medium→hard 구간은 유형 A 가 아니라 사실상 상수공간+공개량. easy→medium 만 카디널리티가 증가(604,800→1,814,400).

**⑥ 난이도 결정축**: 자릿수 N(7/8/8) + 공개량(reveal). N 은 *절벽형* 레버(5D≈96/6D≈89/7D≈62/8D≈28%)라 +1 자리로 과하게 어렵게 한 뒤 reveal 로 완화 — 8D 에선 숫자집합 공개는 무효, **위치 공개가 강력**(≈+28%p/위치, `:190`).

## 1.2 SAT Puzzle (불리언 충족) — `sat_puzzle_en.py` / `_ko.py`

**① 문제 정의 / 정답 형식**
V개 불리언 변수에 대해, 자연어로 표현된 CNF 제약을 모두 만족하는 유일 진리값 할당 탐색. 정답 = `{var: True/False, ...}` (`:55`).

**② 생성 파이프라인**
1. 무작위 해 생성: V개 변수 선택 후 각 True/False 무작위 (`:294-305`).
2. backward clause 생성: 해를 만족하면서 유일해가 될 때까지 CNF clause 추가 (`:307-337`).
3. clause 메커니즘: 리터럴 무작위 선택하되 target 해 하에 최소 1개 True 보장, 중복 제거 (`:348-377`).
4. 유일해 검증: `2^V` 전수 열거, 정확히 1개만 만족 확인 (early-stop=2, `:389-422`).
5. 자연어 변환: CNF → 영/한 문장 (`:424-503`).

**③ 정답 도출 · 유일해 검증 알고리즘**
Brute-force 2^V 열거. 각 할당마다 모든 clause(AND 연결)의 리터럴 중 ≥1 True 확인 (`:398-403`). 유일성 = `_count_solutions(..., stop_after=2)==1` (`:410-422`). 복잡도 O(2^V·|clauses|), V≤14 → 최대 ~2.8M 연산. EN/KO 동일.

**④ 난이도 설정** (`:258-292`, EN/KO 동일)

| 파라미터 | easy | medium | hard |
|----------|------|--------|------|
| num_vars V | **9**(고정) | 11 | 14 |
| min/max_clauses | 36/**54** | 56/88 | 115/170 |
| clause_length | (3,4) | (3,4) | (3,4) |
| negation_ratio | **0.51** | 0.54 | 0.57 |

**⑤ 탐색공간 분석**
정답 = 진리값 벡터 → **2^V**

| 난이도 | V | 가짓수 |
|--------|---|--------|
| easy | 9 | **512** |
| medium | 11 | **2,048** |
| hard | 14 | **16,384** |

clause 수 증가는 후보를 유일해로 좁히는 역할. 후보 공간 크기는 2^V 가 결정.

**⑥ 난이도 결정축**: 변수 수(9/11/14) + 절 수(36–54/56–88/115–170) → 할당공간 × 제약 밀도.

## 1.3 Logic Grid (아인슈타인 퍼즐) — `logic_grid_en.py` / `_ko.py`

**① 문제 정의 / 정답 형식**
N명 × K카테고리에서 각자 카테고리별 1개 값을 1:1 보유하도록 하는 완전 할당 탐색. 정답 = `{person:{category:value,...},...}` JSON (`:39,86-91`).

**② 생성 파이프라인**
1. 해 무작위 생성: N명 샘플, 카테고리마다 K값 샘플, 1:1 배정 (`:256-276`).
2. 직접 제약("앨리스=개") + 간접 제약("빨간집=커피") 합성 (`:316-440`). 목표 제약수 easy 18–20 / medium 18–20 / hard 45–48.
3. 휴리스틱 검증 + 재시도 (`:177-180,442-478`).

**③ 정답 도출 · 유일해 검증 알고리즘**
Backward generation — 무작위 해에서 제약을 역합성 (`:171-175`).

> ⚠️ **유일해가 실제로 검증되지 않음 (코드 직접 확인, 논문 주의 필수).** 생성부(`:178`)가 호출하는 유일성 체크 `_verify_unique_solution` 의 본문은 **`return len(constraints) >= total_facts * 0.6`** 단 한 줄 — 제약 *개수*가 전체 사실(N×K)의 60% 이상인지만 본다 (`:442-478`). 실제 제약을 풀어 해가 1개인지 확인하는 **백트래킹/제약전파 솔버는 코드에 존재하지 않는다**(전 파일 grep으로 다른 솔버 부재 확인). 코드 주석도 이를 자인: "this is a simplified check — in production, you'd use a full CSP solver"(`:451`), "A full implementation would use AC-3 or similar"(`:455`), "In a real implementation, we'd use backtracking to verify"(`:474`).
> → **실측 결과 (정량 확정)**: 생성기를 계측해 구조화 제약을 뽑고 **실제 백트래킹 CSP 솔버**(`validators/check_logic_grid_uniqueness.py`)로 해 개수를 세었더니, **비유일(≥2해) 비율 = easy 99.7% / medium 100% / hard 100%** (각 N=300, 0 aborted). 즉 **사실상 모든 logic_grid 퍼즐이 다중해**다. 독립 검증기로 솔버가 찾은 제2의 해가 17개 제약을 전부 만족함을 확인(over-count 아님). 원인: 제약이 무작위로만 추가돼 자유 셀(예: 미언급 Hobby/Pet)이 자유 치환됨.
> → **결론**: docstring(`:6` "CSP solving with backtracking")·README("Unique Solution Guaranteed / Constraint Propagation으로 검증")는 **사실과 다름**(미구현 + 실측 비유일 ~100%). 논문에서 logic_grid 를 "유일해 보장"으로 기술 **금지**. 평가 시에도 단일 정답 채점은 부당(모델이 다른 유효해를 내면 오답 처리됨) → 데이터셋 재설계(진짜 유일해 검증) 또는 채점 방식 변경 필요.

복잡도: 검증부는 O(제약수·N·K) 문자열 파싱뿐.

**④ 난이도 설정** (`:221-252`)

| 파라미터 | easy | medium | hard |
|----------|------|--------|------|

| 제약 범위 | **18–20** | 18–20 | **45–48** |
| 직접 제약 | 6 | 4 | **11** |

**⑤ 탐색공간 분석**
정답 = 전 카테고리 할당. K개 중 1개를 사람축으로 고정하면 자유도 (K−1) → **(N!)^(K−1)**

| 난이도 | N | K | 공식 | 가짓수 |
|--------|---|---|------|--------|

**⑥ 난이도 결정축**: 격자 크기 (N,K) 증가 → (N!)^(K−1) 지수적 증가 + 제약/직접제약 수.

## 1.4 Minesweeper (지뢰찾기) — `minesweeper_en.py` / `_ko.py`

**① 문제 정의 / 정답 형식**
R×C 격자에 지뢰 M개, 일부 셀이 인접지뢰수로 공개됨. 모든 지뢰 위치 탐색. 정답 = 정렬된 (r,c) 좌표 리스트 문자열 (`:203-204`).

**② 생성 파이프라인**
1. 지뢰 마스크 생성: 전체 셀 중 M개 무작위 (`:784-840`).
2. 공개셀 선택(난이도별): easy=고정수 블록 (`:549-561`); medium/hard=`until_unique` 비율 공개 (`:563-579`).
3. 유일해까지 공개셀 추가 (`solve_puzzle` 최대 2해, `:799-806`).
4. 정보점수 기반 공개 순서 정렬 (`:703-737`).

**③ 정답 도출 · 유일해 검증 알고리즘**
Backward — 지뢰 마스크 먼저, 힌트는 역산. 검증 = **정적 제약-degree 정렬 backtracking + 전역 지뢰수 제약**: 미지정 셀을 *생성 시점에 한 번* 제약수 많은 순으로 정렬(`unknown_cells.sort(key=lambda rc: -len(constraints_per_cell[rc]))`, `:130`) — 동적 MRV(매 단계 재선택)가 아닌 정적 degree 순서. 광산/안전 시도마다 단일단서 제약(`assigned≤v ∧ assigned+unknown≥v`, `:132-145`)+전역 지뢰수 제약(`:147-159`) 검사, `max_solutions=2` 조기종료(유일성=해 1개, `:799-806`). 복잡도 최악 O(2^unrevealed), 제약 가지치기로 단축. *(코드 직접 확인: `solve_puzzle` `:94-187`.)*

**④ 난이도 설정**

| 난이도 | 격자 | 지뢰 M | 공개 | 근거 |
|--------|------|--------|------|------|
| easy | 7×7~12×12 (5블록) | 6~24 | 고정 24~48셀 | `:549-561` |
| medium | 9×9 | 14 | 비율 0.25~0.45 (until_unique) | `:563-570` |
| hard | 9×9 | 18 | 비율 0.25~0.58 (until_unique) | `:572-579` |

> 📌 2026-06: **minesweeper_ko 가 전면 재작성되어 EN 과 수치 완전 동일**(과거 KO 는 5×5/6×6/8×8·mine_ratio 기반의 다른 설계였음). 구버전의 `min_solver_nodes`(0/25/120) 난이도 레버는 **제거**됨(0/0/0). 단 EN 은 easy 파일의 블록별 `difficulty` 필드를 "medium/hard" 로 라벨링하는 반면 KO 는 출력파일명(file_key)을 따라 전부 "easy" 로 둠 — 수치 아닌 메타 라벨 차이.

**⑤ 탐색공간 분석**
정답 = 비공개셀에 M개 지뢰 배치 → **C(R·C − 공개셀, M)**. (공개비율은 *안전셀* 기준 → 비공개셀 = 81 − 공개수, 추정 범위로 보고)

| 난이도 | 격자 | M | 비공개(대표) | 공식 | 가짓수(대표) |
|--------|------|---|--------------|------|--------------|

| easy(대블록) | 12×12 | 24 | 96 | C(96,24) | **~10²⁰** |

출력 좌표공간 = (r,c) 정렬 리스트, 9×9 최대 81좌표. (medium/hard 비공개셀 수는 **데이터셋 100건/티어 `#` 카운트 실측**: medium 평균 59.9, hard 56.0.)

**⑥ 난이도 결정축**: 격자 크기 × 지뢰밀도(M/셀) × (1 − 공개비율) → 제약 추론 강도.

## 1.6 Inequality (부등식 배치) — `inequality_en.py` / `_ko.py` ※ 유형 A*(탐색공간 비단조·혼합) · **EN=KO 직번역**

> ⚠️ **분류 (코드 재확인, 2026-06 개정 반영)**: 본 task 는 후보 공간이 유한하나 **"공간=난이도(유형 A)" 가 성립하지 않는다** → 유형 A*.
> - **EN=KO 동일(직번역)**: 과거엔 KO 가 전 난이도 1D 체인이었으나, **KO 생성기가 EN 과 동일하게 전면 재작성**되어 이제 두 언어가 `EASY_CHAIN_BANDS`·`FUTOSHIKI_TIERS`·`FUTOSHIKI_GIVENS_BY_SIZE` 까지 **byte 단위로 일치**한다(검증: `diff` 결과 동일, `inequality_en.py:936-952` ≡ `inequality_ko.py:911-926`). 즉 직전 버전 문서의 "KO 비단조(medium>hard), visible solver-step" 서술은 **구버전 KO 코드 기준이라 무효**.
> - **구조 전환(1D→2D)**: easy=1D 부등호 체인, medium/hard=2D Futoshiki(라틴방진). tier 경계에서 문제 종류·정답 형식이 바뀌므로 단일 "size! 공간" 축으로 비교 불가.
> - **비단조**: easy 큰 밴드(size **16**, hide 0)는 nominal size! 이 16!≈**2.1×10¹³** 로 medium/hard Futoshiki 라틴방진(5×5=1.6×10⁵, 6×6=8.1×10⁸)보다 **훨씬 크지만 가장 쉽다**. medium↔hard 는 격자 집합 {5,6} 이 **동일**하고 차이는 6×6 비중(38%→80%)·backtrack_ratio(0.82→0.92)뿐 → 카디널리티 점프가 아니라 **backtrack 비율**이 난이도 driver.
> → 아래 ①~④ 는 EN/KO 공통, ⑤⑥ 의 "탐색공간=난이도" 해석은 위 정정을 따른다.

**① 문제 정의 / 정답 형식**
easy=1D 부등호 체인, medium/hard=2D Futoshiki(라틴방진+부등호). 정답 = 1D는 1..size 순열 인코딩(size≤9 숫자, >9 base-36; size 16 은 base-36 단문자, `:98-100`), Futoshiki는 n² 숫자 공백구분 (`flat_answer` `:506-509`). EN/KO 동일.

**② 생성 파이프라인** (EN/KO 동일)
- *1D Easy*: ① 1..size 무작위 순열 ② 인접쌍 비교로 부등호 도출 ③ 유일해까지 힌트 추가 ④ 유일성 유지하며 부등호 숨김(num_to_hide 0 또는 1) ⑤ 힌트 최소화 (`generate_chain` `:308-`).
- *Futoshiki*: ① 라틴방진 생성(행/열 셔플 재명명) ② 전체 인접부등호 도출 ③ 힌트 노출(grid크기별 givens) ④ 유일성 유지 제약 최소화 ⑤ propagation/backtrack 모드 분류(`backtrack_ratio` 확률로 target_mode 결정) (`FutoshikiGenerator.generate` `:703-`).

**③ 정답 도출 · 유일해 검증 알고리즘**
**Domain-minimization MRV backtracking + greedy 부등호 숨김**(1D), **capped backtracking 해 카운트**(Futoshiki). 유일성 = `max_count=2` 솔버로 해≤2 확인. `MIN_CONSTRAINTS=3`. 복잡도 최악 O(size!)(1D) / O((n!)^n)(Futoshiki). EN/KO 동일.

**④ 난이도 설정** (`EASY_CHAIN_BANDS` `:911-915`, `FUTOSHIKI_TIERS` `:919-922`, `FUTOSHIKI_GIVENS_BY_SIZE` `:926`; EN/KO 완전 동일)

| 난이도 | 타입 | 크기(가중) | 비고 |
|--------|------|------------|------|
| easy | 1D 체인 | [8,9] hide0(35%) / [8,9] hide1(50%) / **[16]** hide0(15%) | size ∈ {8,9,16} |
| medium | Futoshiki | 5×5(62%) / 6×6(38%) | backtrack_ratio 0.82; givens 5×5=2–6, 6×6=3–7 |
| hard | Futoshiki | 5×5(20%) / 6×6(80%) | backtrack_ratio 0.92; givens 5×5=2–6, 6×6=3–7 |

> EN easy 큰밴드 크기는 구버전 [20,28] → **[16]** 으로 축소됐고 hide-0 [8,9] 밴드가 추가됨(`git diff`). medium/hard 는 EN/KO 모두 불변.

**⑤ 탐색공간 분석** (⚠ 구조 전환 + 비단조 — 위 분류 박스 참조)
- 1D 체인: 정답 = 1..size 순열 → **size!**
- Futoshiki n×n: 행=순열 라틴방진 → 상한 **(n!)^n**, 정확 라틴방진 수 5×5=**161,280** / 6×6=**812,851,200**

| 난이도 | 공식 | 가짓수 | 난이도와 단조? |
|--------|------|--------|----------------|
| easy | size! (1D) | 8!=**40,320** / 9!=**362,880** / 16!≈**2.1×10¹³** (큰밴드, hide 0 → **가장 쉬움**) | — (1D) |
| medium | 라틴방진 (2D) | 5×5=**161,280** ~ 6×6=**8.1×10⁸** (상한 (n!)^n: 2.5×10¹⁰ / 1.4×10¹⁷) | 구조 전환(1D→2D) |
| hard | 라틴방진 (2D, 6×6 비중↑) | 6×6 중심 → **8.1×10⁸** (격자집합 medium과 **동일**) | backtrack 비율↑(0.82→0.92) |

→ easy 큰밴드(16!≈2.1×10¹³)가 medium/hard Futoshiki 카디널리티(≤8.1×10⁸)보다 **크지만 가장 쉽다** → size! 는 난이도 대리지표로 부적합. medium↔hard 는 같은 격자집합(카디널리티 동일)이고 차이는 **backtrack 비율**. **"탐색공간=난이도" 불성립.**

**⑥ 난이도 결정축**: 탐색공간(size!/라틴방진)은 난이도와 단조 아님 → 결정축은 카디널리티가 **아니다**. **문제 구조 전환**(easy 1D → medium/hard 2D Futoshiki) + **backtrack 비율**(0.82→0.92)이 지배축. ⇒ 유형 A 형식이나 실질은 유형 C(추론 깊이)에 가까운 A* 사례. EN/KO 동일(직번역).

## 1.7 Kinship (친족 호칭, KO) — `kinship.py` ※ **유형 B (재분류, num_choices 26 고정)**

## 2.1 Cryptarithmetic (복면산) — `cryptarithmetic_en.py` / `_ko.py` ※ **2026-07 3독립표(글자그룹/자모) 전면 재작성 · 유형 A(부분)**

> 📌 **2026-07 재작성 (커밋 `5c3ec83`, 재측정 완료)**: 구버전은 **단일 10-글자 치환표**(모든 글자가 서로 다른 숫자 → 10! = 3,628,800 상수 공간, 피연산자 2/3/4)였다. 재작성판은 **실제 단어 + 3개 독립 치환표**다:
> - **EN**: 알파벳을 3개 고정 그룹으로 나눔 — `g0=A–I` / `g1=J–R` / `g2=S–Z`. 각 그룹이 **자체 독립 표**(글자→숫자 0–9, 그룹 안에서만 서로 다른 글자=서로 다른 숫자). 세 표는 독립(g0 의 A=5 이어도 g2 의 S=5 가능).
> - **KO**: 각 한글 음절을 초성/중성/종성 분해 → **초성표·중성표·종성표 3개 독립**(cipher_ko 자모 분해 재사용). 받침 없는 음절은 초성·중성 2자리.
> - 두 언어 모두 **피연산자·결과가 실제 단어**, 각 단어 첫 자모/글자 ≠ 0, 정답 = 결과 단어의 숫자값.
> - **유일성은 "일부 공개된 부분키(revealed)" 조건부**: 키를 일부 공개하고 나머지는 열별 덧셈 제약으로 추론(`_select_revealed` 가 유일성 유지 선까지 가림). ⚠ **공개키가 없으면 비유일**(실측: EN easy 첫 3건 모두 공개키 없이는 해 ≥5, 공개키 조건부 해 = 1).
> → 구버전의 "10! 상수 공간 / 피연산자 2·3·4 / target_letters=(10,10)"은 **전부 무효**. 아래 ①~⑥ 은 재작성판·재측정 기준.

**① 문제 정의 / 정답 형식**
피연산자 4개(전 난이도 고정) + 결과 1개, 모두 실제 단어. 각 글자/자모는 소속 그룹(EN 3글자그룹 / KO 초·중·종성)의 표에 따라 숫자. 정답 = 결과 단어의 숫자값(정수). 문제에 **부분키가 공개**되고 나머지는 추론 (EN `:97-115,325-403`, KO `:392-`).

**② 생성 파이프라인** (EN `:335-403` / KO `:402-`)
1. 실제 단어 풀에서 자모/글자 길이 L(=자릿수)인 피연산자 4개 + 결과 단어(길이 L 또는 L+1) 샘플링 (EN `:352-364`).
2. 그룹별 독립표 배정이 존재(덧셈 성립)하는 해를 `find_solutions(max_count=2, max_nodes=30000)` 로 탐색 (`:367`).

**③ 정답 도출 · 유일해 검증 알고리즘**

# Part 4. 탐색공간과 gemini-3-flash 정확도의 관계 — 가설 검정

> ⚠️ **2026-06/07 개정 주의 (Part 4~5 한정)**: Part 4~5 는 **구버전 config** 로 작성된 정성 분석이다. 일부 task 의 config-값 인용(number_baseball **3/5/6**, inequality **KO 비단조/visible solver-step**, kinship **8/12/20**, **cryptarithmetic 10! 상수·피연산자 2/3/4**)은 **현재 코드와 다르며 Part 1~3 이 최신**이다. 특히 **cryptarithmetic 은 2026-07 재작성으로 3독립표·유형 A(부분)** 이 됐다(§2.1 실측). 아래에서 직접 모순되는 값은 정정했으나, 정성 결론("raw 탐색공간 ≠ 난이도")은 유지되며 **Part 6 의 실측 회귀(R²=0.974 tier)가 이를 더 강하게 지지**한다. 측정 정확도가 필요한 곳은 Part 6 의 `accuracy_per_task.json` 을 1차 출처로 삼는다.

## 4.0 전제: 관찰된 정확도 밴드

세 난이도에서 gemini-3-flash-preview(reasoning=medium)의 실측 정확도는 전 모듈에서 다음 밴드에 든다:

- **easy: 70–80%**, **medium: 40–60%**, **hard: 20–30%**

이는 코드의 보정 목표(easy~75% / medium~50% / hard~25%; hanoi 는 2026-07 개정으로 hard 목표 ~20%, `array_formula:11-14`, `cipher:23-43`, `hanoi:196-200`, `logic_grid:204`, `saju:5`, `jamo:10-11` 등)와 정확히 일치한다. 즉 **난이도는 이 정확도 밴드로 *조작적 정의*(operationally defined)** 되어 있다.

## 4.1 검정할 가설

> **H (탐색공간 가설)**: 탐색공간(정답 후보 가짓수)이 클수록 gemini-3-flash 정확도가 낮아지고, **이 관계가 모듈을 가로질러 일관되게** 나타난다. → 탐색공간이 난이도(정확도)의 공통 설명 변수다.

이 가설을 (a) 모듈 *내부*(within-module), (b) 모듈 *간*(across-module) 두 수준에서 검정한다.

## 4.2 모듈 내 검정 — 부분적으로만 성립 (그나마 설계상 자명)

탐색공간이 난이도 knob 인 **유형 A 모듈에서는** 난이도가 오르면 탐색공간도 커지고 정확도는 떨어진다 — 그러나 이는 *발견*이 아니라 **그렇게 되도록 파라미터를 보정**한 결과다.

| 모듈(유형 A) | easy 공간→정확도 | medium | hard |
|---|---|---|---|
| sat | 2⁹–²·¹⁰ → ~75% | 2¹¹ → ~50% | 2¹⁴ → ~25% |
| logic_grid | 2.1×10⁸ → ~75% | 2.7×10¹¹ → ~50% | 4.3×10²⁷ → ~25% |
| sudoku(9^K) | 6,561 → ~75% | 59,049 → ~50% | 531,441 → ~25% |

**그러나 유형 A(및 A*) 안에서도 반례 존재** (현재 코드 기준):
- **number_baseball (medium=hard 동일공간, reveal 변별)**: 현 config 는 N 7/8/8 → easy P(10,7)=**604,800** < medium=hard P(10,8)=**1,814,400**. 즉 **medium 과 hard 의 nominal 공간이 완전히 같은데**(둘 다 8자리) 실측 정확도는 medium 0.52 vs hard 0.26 (`accuracy_per_task.json`)으로 갈린다 — 공간이 아니라 **reveal**(medium 0.6 위치공개 vs hard 무공개)이 난이도를 가른다. "공간↑⇒정확도↓"가 medium→hard 구간에서 성립 안 함. *(구버전 3/5/6 시절 gpt-5.4-mini 에서 관찰된 "easy>medium 역전"은 구 config/구 데이터 기준이라 폐기.)*
- **inequality (구조 전환, 유형 A*)**: easy=1D 체인(큰밴드 16!≈**2.1×10¹³**) ↔ medium/hard=2D Futoshiki(라틴방진 ≤**8.1×10⁸**). **easy 의 큰 인스턴스가 medium/hard 보다 nominal 공간이 크지만 가장 쉽다**(easy 0.84 > medium 0.48 > hard 0.23, `accuracy_per_task.json`). tier 마다 문제 구조가 바뀌어 size! 비교 자체가 부적절 → 난이도는 구조 전환 + backtrack 비율(0.82→0.92). EN=KO 동일(직번역).
- **cryptarithmetic (medium≈hard 동일공간, 가림량 변별, 유형 A부분, 2026-07 재작성)**: EN raw 공간이 easy 10¹²·⁵ → **medium 10¹⁴·⁴ ≈ hard 10¹⁴·²**(둘 다 ~10¹⁴, 오히려 medium 근소히 큼)로 medium→hard 가 사실상 동일. 즉 nominal 공간이 medium/hard 를 못 가르며, 난이도차는 **공개키 가림량(추론 깊이)** — 유효 backtrack 노드 medium 2,034 < hard 10,966·미공개 토큰 10.5 < 12.2(§5.2b 실측). ⚠ **단 crypt 는 `accuracy_per_task.json` 에 없어 *측정 정확도*로는 확인 불가**(공간·유효노드 사실까지만; number_baseball 처럼 정확도 split 을 대지는 못함).

→ 모듈 내에서도 "공간↑ ⇒ 정확도↓"는 **단일 파라미터(자릿수·구조)에선 깨질 수 있다**. 후보 가짓수보다 *제약·reveal·구조가 추론을 얼마나 요구하는가*(유효 분기·구조)가 지배적이기 때문.

## 4.3 모듈 간 검정 — 명백히 기각

같은 정확도 밴드(예 easy ~75%)가 **탐색공간 60 ~ 10²⁷ 에 걸쳐** 흩어져 있다. 정확도가 ~75%로 같은데 공간은 25자릿수 차이가 난다 → 탐색공간은 모듈 간 정확도의 예측자가 **아니다**.

| easy(~75% 정확도) 모듈 | easy 탐색공간 |
|---|---|
| saju(기둥) | **60** |
| number_baseball | 604,800 (P(10,7)) |
| sudoku(9^K) | 6,561 |
| cipher | ≤9 (키후보) |
| cryptarithmetic (2026-07 재작성) | **~3.3×10¹²** (EN ∏P(10,kᵍ)) |
| jamo | 1.25×10⁸ |
| logic_grid | 2.1×10⁸ |
| yacht | **4.79×10⁸** (12!) |

→ 정확도 ~75% 고정, 공간은 60 → ~10¹². **무상관.** (crypt easy 공간은 구 10!→재작성 후 ~10¹²·⁵ 로 커져 범위가 더 넓어짐.)

## 4.4 결정적 반례 — 유형 B (탐색공간 *상수*인데 정확도는 변함)

탐색공간이 난이도와 **무관하게 일정**한데도 정확도는 80→50→20%로 떨어지는 모듈들 → 가설을 within-module 에서까지 직접 반증한다.

| 모듈 | easy→hard 탐색공간 | easy→hard 정확도 | 실제 난이도 knob |
|---|---|---|---|
| **jamo** | easy=medium **1.25×10⁸ 동일**, hard 1.39×10¹² | ~80→**45**→25% (`:11,39`) | 받침 복잡도(light→single→겹받침) |
| **yacht** | 12! = 479,001,600 **(상수)** | 75→50→25% | 결정복잡도(top1−top2 점수마진) |

특히 **jamo의 easy와 medium은 탐색공간이 완전히 동일**(11,172²)한데 정확도가 ~80% vs 45% 로 갈린다 — 공간이 아니라 **종성(받침) 처리 난도**가 정확도를 움직인다는 직접 증거.

> ⚠️ **cryptarithmetic 은 이 표에서 제거됨(2026-07 재작성)**: 구버전은 10! 상수 공간이라 여기 대표 반례였으나, 재작성판은 탐색공간이 상수가 아니다(∏P(10,kᵍ), EN 10¹²·⁵→10¹⁴·⁴, KO 단조 증가) → 유형 A(부분)로 이동(§2.1). 단 **EN medium≈hard(둘 다 ~10¹⁴)** 구간은 여전히 "공간 고정, 정확도 변동" 성격이라 §4.2 number_baseball 형 부분 반례로 남는다.

## 4.5 유형 C — 가설이 적용 불가

array_formula·causal_dag·ferryman·hanoi 는 정답이 유일 계산값이라 **유한 탐색공간 자체가 없다**. 그럼에도 75/50/25% 밴드에 정렬된다 → 정확도는 후보 가짓수와 **무관하게** 추론 깊이(연산 수·그래프 크기·시퀀스 길이)로 조절된다.

## 4.6 결론 — "일관된 양상"의 정체

- **탐색공간 가설(H)은 강한 형태(모듈 간 일관)로 기각된다.** 탐색공간 크기는 gemini-3-flash 정확도의 *필요조건도 충분조건도 아니다*: 같은 정확도가 60~10⁸ 공간에 걸쳐 나오고(4.3), 상수 공간에서 정확도가 변하며(4.4), 공간이 없어도 밴드에 맞고(4.5), 공간↑이 정확도↑인 경우도 있다(4.2 number_baseball).
- **실제로 일관된 단 하나의 양상**은 탐색공간이 아니라 **"gemini-3-flash 정확도 70–80 / 40–60 / 20–30% 밴드"** 그 자체다. 17개 task 가 *서로 다른 knob*(탐색공간 / 제약 밀도 / 받침 / 지식 / 결정 마진 / 추론 깊이)로 **동일한 정확도 밴드에 수렴**하도록 보정됐다. 공통점은 출력(정확도)이지 입력(탐색공간)이 아니다.
- **탐색공간이 정확도를 예측하는 듯 보이는 경우**는 오직 "보정 knob 가 마침 탐색공간 크기였던 일부 유형 A 모듈"에 한정되며, 그조차 number_baseball 에서 깨진다.

## 4.7 가설을 살리려면 — 올바른 축 (대안 가설)

탐색공간(raw 후보 수) 대신 다음이 정확도와 더 일관되게 연결될 가능성(미검증, 추가 측정 필요):
1. **유효 탐색공간(effective)** = 제약·힌트로 가지치기한 *뒤* 남는 분기 수(= 솔버 backtrack 노드 수). raw 공간이 같아도 제약 구조가 다르면 이 값이 달라져 number_baseball·jamo 반례를 설명할 수 있음.
2. **추론 단계 길이(serial reasoning depth)**: hanoi k, cryptarithmetic 자릿수×가림량(미공개 추론 토큰 수), causal_dag 경로 홉 — 순차 의존 단계 수.
3. **지식 의존도**: saju 일주처럼 *추론으로 복원 불가한* 외부 데이터 요구 여부 (능력 공백 유발).

→ 논문 주장: **"난이도는 raw 탐색공간이 아니라 gemini-3-flash 정확도로 정의되며, 탐색공간은 그 정확도의 일관된 예측자가 아니다. task 별 지배적 기제(유효 분기·추론 깊이·지식)는 다르되, 모두 동일 정확도 밴드로 수렴한다"** — 이것이 데이터(코드 보정 주석 + 탐색공간 3유형)가 직접 지지하는 결론.

## 4.8 데이터 fact / 미검증 / 가설 분리 (CLAUDE.md §10)

1. **데이터로 확정**: 보정 목표 밴드 75/50/25%(주석 인용), 탐색공간 3유형 분류, yacht 상수 공간 + 난이도 증가, jamo easy=medium 동일 공간, number_baseball medium=hard 동일 공간(P(10,8))이고 차이는 reveal, **cryptarithmetic(2026-07 재작성) EN medium≈hard 동일 공간(∏P(10,kᵍ)≈10¹⁴)이고 차이는 공개키 가림량**(§2.1 실측).
2. **미검증 — 추가 측정 필요**: 각 모듈의 *gemini-3-flash 실측* 정확도 원자료(주석은 목표·일부 실측만 기재), §4.7 의 "유효 탐색공간/추론 깊이"가 실제로 정확도와 상관되는지(회귀 분석 필요).
3. **가설(미검증)**: §4.7 의 effective-branching·reasoning-depth 축이 raw 공간보다 정확도를 잘 설명할 것이라는 주장 — 본 문서는 raw 공간 가설의 *기각*까지만 데이터로 지지하며, 대안 축의 우월성은 아직 측정 안 됨.

# Part 5. §4.7 대안가설 검정 — 계산 가능한 범위에서 직접 분석

§4.7 은 raw 탐색공간 대신 ① 유효 탐색공간(제약·힌트로 가지친 분기 수) ② 순차 추론 깊이 ③ 지식 의존도 가 정확도와 더 일관될 것이라 추측했다. 이를 실제로 검정한다.

## 5.0 데이터 가용성 (§18 광범위 검색 결과 — 정직한 한계)

- **측정 정확도: repo에 없음.** `results/` 디렉토리 부재(`find` 0건), task별 accuracy 파일 0건. `predicted_accuracy_graph.png` 는 **측정값이 아니라 설계 목표(75/50/25%) 막대**(cipher 예시)임을 이미지로 확인.
- **per-instance 유효지표: 데이터셋 미저장.** `data/jsonl/*` 필드 = `id/question/answer/solution/difficulty` (sat만 `variables/clauses` 추가). solver_steps/nodes 등은 생성 시 게이팅에만 쓰고 폐기.
- **결론**: "alt-metric vs *측정* 정확도" 회귀는 repo 데이터로 **불가**. 단, alt-metric 을 데이터셋에서 *재계산*해 **(a) 난이도와 단조인지 (b) raw-space 의 역전/동률을 해소하는지**(= 대안가설의 *필요조건*)는 측정 정확도 없이도 검정 가능 → 아래 수행.

## 5.1 jamo — 대안가설 **지지** (② 추론깊이 프록시 = 받침 조작 단계)

raw 탐색공간은 easy=medium 동률(11,172²≈1.25×10⁸). 데이터셋 100건/티어에서 답 음절을 유니코드 분해해 받침(종성≠0)·겹받침 수를 직접 계산:

| tier | n_syl | 받침수/건 | 받침비율 | 겹받침수/건 | raw 공간 |
|------|------|-----------|----------|-------------|----------|
| easy | 2.00 | **0.67** | 0.34 | 0.17 | 1.25×10⁸ |
| medium | 2.00 | **1.88** | 0.94 | 0.24 | 1.25×10⁸ (=easy) |
| hard | 3.00 | **2.85** | 0.95 | 0.81 | 1.39×10¹² |

→ raw 공간이 easy=medium 동률인 지점에서 **받침 조작 단계는 0.67 < 1.88 (2.8×) 로 분리**되고 전체적으로 단조(0.67<1.88<2.85). 겹받침도 0.17<0.24<**0.81**(hard 급증). **대안가설의 필요조건 충족** — 유효 추론깊이가 raw 공간의 동률을 해소하고 난이도 순서와 정렬. *(단조성·동률해소까지만 확인; 측정 정확도와의 상관은 데이터 부재로 미검증.)*

## 5.2 number_baseball — 대안가설 **불충족** (① 유효 잔여 모호도)

> ⚠️ **본 §5.2 는 구버전 config(N 3/5/6, 힌트 3/3/5)·구버전 데이터 기준이라 수치가 모두 superseded.** 현 코드는 N **7/8/8** + reveal(easy 숫자5공개·medium 위치0.6) 이고, 데이터도 재생성됨(`git diff` 에 number_baseball jsonl 변경 확인). 따라서 아래 표(720/30240/151200, easy 1.10 등)와 "데이터 stale(3 vs 4 힌트)" 결론은 **현 데이터에 해당하지 않는다**(재생성으로 해소). 현 config 의 유효지표를 보려면 §12.2 솔버를 재생성 데이터에 재실행해야 함(미수행). Part 6 의 실측 정확도(number_baseball_en 0.78/0.52/0.26)가 현 데이터 기준 1차 출처.

코드의 `BullsAndCows.calculate_strikes_balls`/`find_all_solutions` 를 **재사용**(§12.2)해, *구버전* 데이터셋 100건/티어의 힌트로 후보 잔여수를 재계산(아래는 구 데이터 기록):

| tier | N | raw P(10,N) | #hints | 1힌트후 잔여 | 마지막직전 잔여 | 최종 |
|------|---|-------------|--------|--------------|------------------|------|
| easy | 3 | 720 | 3 | 4.0 | **1.10** | 1.0 |
| medium | 5 | 30,240 | 3 | 45.0 | **15.0** | 1.0 |
| hard | 6 | 151,200 | 5 | 264 | 14.1 | 1.0 |

문제의 반례: gpt-5.4-mini 에서 easy(84%) < medium(96%) — easy 가 *더 어려웠다*(`number_baseball_ko:68`). 그러나 유효 잔여 모호도는 **easy 1.10 < medium 15.0** — 즉 유효지표는 medium 을 *더 어렵게* 본다. **raw 공간(medium 큼)과 같은 방향**이며, 측정된 역전을 **설명하지 못한다**.

→ **§4.7 의 ①(유효 탐색공간)은 number_baseball 반례를 해소하지 못함.** 오히려:
- easy 는 마지막 힌트 직전 이미 잔여 ~1.1 (거의 결정됨) 인데도 모델이 ~16% 틀림 → 난이도는 *탐색 모호도*가 아니라 **연역/산술 실행 오류**에서 옴(가설, 미검증).
- 해당 측정은 gpt-5.4-mini·소표본(다른 주석 "20-sample") 이라 **모델 특이·표본 잡음** 가능성(가설). gemini-3-flash 재측정 필요.
- ⚠️ **데이터 stale 확정**(재생성으로 검증): 현재 코드는 easy 4힌트 생성(`ProblemGenerator.generate_problem`, seed 0, 30/30 = 4힌트), 그러나 데이터셋 easy 는 3힌트(100/100). 즉 **공개 데이터셋이 구버전 config(min_hints=3) 산출물** — 현 코드(min_hints=4)와 불일치. → 정확도 측정 전 **데이터 재생성 필수**(안 그러면 더 쉬운 구버전 퍼즐을 측정).

## 5.2b cryptarithmetic — 대안가설 **지지** (① 유효 backtrack 노드) ※ **2026-07 재작성 재측정**

> 📌 **재측정(2026-07 재작성판)**: 구버전 수치(raw 10! 상수, 유효 노드 10.7K/21.6K/31.2K)는 **무효**. 재작성판은 raw 공간이 상수가 아니고(∏P(10,kᵍ)), 유일성이 **공개키(revealed) 조건부**라 유효 노드는 **공개키를 고정한 채** 측정해야 한다. 코드의 `find_solutions(fixed=revealed, _stats=)` 를 재사용해 데이터셋 100건/티어(EN+KO)의 등식+공개키를 파싱·재실행 (`measure_cryptarithmetic.py`):

| 언어 | tier | 자릿수 | raw 공간 ∏P(10,kᵍ) log10(중앙) | 미공개(추론) 토큰수 | **유효 backtrack 노드**(공개키 조건부, median/mean/max) |
|------|------|--------|-------------------------------|--------------------|-------------------------------------------------------|
| EN | easy | 6 | 12.52 | 8.8 | **480** / 1,085 / 8,398 |
| EN | medium | 7 | 14.42 | 10.5 | **2,034** / 10,122 / 144,674 |
| EN | hard | 8 | **14.23**(med≈hard) | 12.2 | **10,966** / 57,793 / 698,928 |
| KO | easy | 6 | 14.42 | 7.7 | **83** / 270 / 6,298 |
| KO | medium | 7 | 15.37 | 10.3 | **478** / 1,433 / 30,022 |
| KO | hard | 7–8 | 16.04 | 12.3 | **4,104** / 12,175 / 146,327 |

→ **핵심 관찰**: raw 공간이 **EN 에서 med≈hard 로 비단조**(14.42→14.23, 오히려 medium 근소히 큼)인데도 유효 backtrack 노드는 **480<2,034<10,966(EN)·83<478<4,104(KO)로 깨끗이 단조**. 즉 **raw 공간이 medium/hard 를 못 가르는 지점에서 effective(유효 노드·미공개 토큰수)가 난이도를 분리** — 대안가설 ① 지지(단조 + raw 비단조 보완). 미공개(추론) 토큰수도 8.8<10.5<12.2(EN)·7.7<10.3<12.3(KO)로 단조. jamo 에 이어 두 번째 지지 사례. *(단조성·비단조 보완까지만 확인; 측정 정확도와의 상관은 accuracy 파일에 crypt 부재로 미검증.)*

## 5.3 종합 — 대안가설은 *부분적으로* 성립 (3건 중 2건 지지)

| 모듈 (유형) | 검정한 alt-metric | raw 공간 | 결과 |
|------|-------------------|----------|------|
| jamo (B) | ② 받침 조작 단계 | easy=medium 동률 | ✅ 동률 해소 + 단조 (0.67<1.88<2.85) |
| cryptarithmetic (A부분, 2026-07 재작성) | ① 유효 backtrack 노드 | EN med≈hard 동률(~10¹⁴) | ✅ 비단조(med≈hard) 보완 + 단조 (EN 480<2,034<10,966; KO 83<478<4,104) |
| number_baseball (A) | ① 유효 잔여 후보 | easy<medium | ❌ 역전 미해소 (단, **데이터 stale**: §5.2 주, 현 코드는 easy 4힌트지만 데이터 3힌트) |

→ **raw 탐색공간이 상수/동률/비단조인 모듈(jamo 동률·crypt EN med≈hard)에서는 effective/추론깊이 지표가 난이도와 단조로 정렬** — §4.7 의 ①②가 raw 가 설명 못 하는 부분을 메운다는 **필요조건을 2건에서 충족**. 유일한 불충족 사례(number_baseball)는 데이터 staleness(3 vs 4 힌트) + 실행오류 가능성이 얽혀 있어 깨끗한 반례가 아니다. → Part 4 의 "raw 공간은 정확도 예측자 아님"은 유지되며, **effective 지표는 유망하나 측정 정확도와의 *상관*은 여전히 미검증**(데이터 부재).

| hard | 6 | −0.65 | −0.37 | 15.2 자릿수 | 0.10 |

→ 약한 **음의** 경향(공간↑ → 정확도 약간↓)은 있으나: (a) tier간 **비일관**(Spearman −0.20~−0.81), (b) **경제적으로 무의미** — raw_space 가 **11–15 자릿수** 변하는 동안 정확도는 **0.10–0.13**만 변동, (c) n=6·raw_space 추정치라 비강건. → **raw_space 는 정확도의 일관·유의미 예측자가 아니다.**

## 6.4 직접 반례 — Type B (raw 상수인데 정확도는 tier로 완전 변동)

| task | raw_space | easy | medium | hard |
|------|-----------|------|--------|------|
| cipher_en | ~상수(키후보 9) | 0.74 | 0.56 | 0.22 |
| yacht_dice_en | 상수(12!) | 0.75 | 0.46 | 0.21 |

→ raw_space 가 **완전 상수**인데 정확도는 0.75→0.20 풀스윙. 정확도를 움직이는 건 raw_space 가 아님(레이어 깊이·결정복잡도 = tier 보정).

## 6.5 결정적 관찰 — easy 정확도는 raw_space 11자릿수에 걸쳐 평평

cipher(log_raw 0.95)=0.74 ↔ minesweeper(log_raw 12.0)=0.71. **raw_space 11자릿수 변동, easy 정확도 0.71–0.84 평평.**

# Part 7. Effective-metric 회귀 — §4.7 검정 (결론: 이 데이터로는 판별 불가)

§4.7 은 effective(backtrack 노드·추론깊이)가 raw 탐색공간보다 정확도를 잘 설명할 것이라 추측했다. Part 6 이 남긴 잔차(2.6%)를 effective 가 설명하는지 검정했다. (스크립트: `scripts/analysis/analyze_effective_metrics.py`)

## 7.1 통계 검정력 — 잔차가 표본 노이즈 이하 (결정적)

데이터셋은 task·난이도당 **정확히 N=100건**(jsonl 라인 수 확인). 이항 표본오차 SE=√(p(1−p)/100):

| tier | task간 관측 SD | 이항 SE@N100 | 판정 |
|------|----------------|--------------|------|
| easy | 0.046 | 0.042 | 신호 ≈ 노이즈 |
| medium | 0.030 | 0.050 | **신호 < 노이즈** |
| hard | 0.031 | 0.042 | **신호 < 노이즈** |

→ tier 차감 후 남는 task간 변동(SD 0.03–0.046)이 **순수 표본 노이즈(SE 0.042–0.050)와 같거나 더 작다**. 즉 medium·hard 에선 task별 정확도 차이가 표본오차로 설명되는 수준 → **설명할 task-level 신호가 사실상 없다.** 어떤 지표(raw든 effective든)를 끼워도 노이즈를 적합하는 것.

## 7.2 effective 지표는 tier와 공선 (collinear by design)

데이터셋에서 직접 계산한 effective 지표(tier별 median):

| task | easy | medium | hard | 단조? | (정확도) |
|------|------|--------|------|-------|----------|
| hanoi (k=추적 이동수, **2026-07 hard 개정**) | 28 | 34 | 45 | ↑ | flash 0.69/0.40/~0.18 |
| number_baseball (1힌트후 잔여) | 2 | 45 | 264 | ↑ | 0.78/0.52/0.26 |
| sat (2^V) | 512 | 2048 | 16384 | ↑ | 0.79/0.52/0.20 |
| causal_dag (이벤트 수) | 28 | 41 | 53 | ↑ | 0.80/0.50/0.24 |
| minesweeper (backtrack 노드) | — | 26 | 2922 | ↑(med·hard) | 0.71/0.51/0.16 |

⚠️ number_baseball(2/45/264) 행은 **개정 이전 데이터**(구 N 3/5/6) 기준 — 재생성/현 config 에선 달라짐(§8 배너). hanoi 행은 **2026-07 재작성·재측정본(k median 28/34/45, hard k42-48 단일 삼중해시)** 으로 갱신. 단조성 결론 자체는 불변.

→ effective 지표는 전부 **tier와 함께 단조 증가**. 이는 effective 가 설계 난이도의 유효 proxy임을 보이지만, 동시에 **raw·effective·tier 가 모두 tier 축으로 공선**임을 뜻한다. 정확도가 tier로 75/50/25 보정됐으므로, 셋 다 정확도와 같은 방향으로 움직이는 게 당연 — 어느 것이 "원인"인지 분리 불가.

## 7.3 결론 — §4.7 은 이 데이터셋으로 답할 수 없다

- **판별 불가 (confounded + underpowered)**: ① 동일 밴드 보정으로 raw/effective/tier 공선, ② tier내 잔차는 N=100 표본 노이즈 이하. 두 이유로 raw vs effective 의 정확도 설명력 우열을 **통계적으로 가릴 수 없다.**
- 단, effective 가 raw 보다 **설계 난이도**(tier)를 잘 추적하는 사례는 이미 확인됨: cryptarithmetic(§5.2b, 2026-07 재작성 — raw EN med≈hard 비단조인데 effective 노드 EN 480→10,966·KO 83→4,104 단조)·jamo(§5.1, raw 동률인데 받침 0.67→2.85 단조). 이는 **raw가 상수/동률/비단조인 곳에서 effective가 난이도를 구분**함을 보이나, 해당 task 들은 accuracy_per_task.json 에 없어 *정확도와의 연결*은 미검증.

## 7.4 §4.7 을 진짜로 검정하려면 (설계 요건)

공선·노이즈 문제를 깨려면:
1. **동일 밴드 보정을 풀고**, effective 를 고정한 채 raw 만(또는 그 반대) 바꾼 셀을 만들어 측정 — 자연 실험.
2. **고정 tier 내에서 effective 를 의도적으로 분산**시킨 변형들을 **큰 N(≥1000)**으로 측정 → 노이즈 바닥을 낮춰 task-level 신호 검출.
3. **raw가 상수/동률/비단조인 task(jamo·yacht·cipher·cryptarithmetic[EN med≈hard])의 정확도를 effective 분위수별로** 측정 — raw가 통제되므로 effective 단독 효과 분리 가능. (이 task들 accuracy 수집이 최우선)

## 7.5 데이터 fact / 미검증 / 가설 (§10)

1. **확정**: N=100/cell, tier내 잔차 SD ≤ 이항 SE(medium·hard에서 엄격히 작음), effective 지표(hanoi k·nb 잔여·sat 2^V·causal_dag 이벤트·minesweeper 노드) 전부 tier 단조.
2. **미검증**: effective 가 raw 보다 정확도를 잘 설명하는지 — 본 데이터로 **판별 불가**(추가 측정 설계 §7.4 필요).
3. **가설**: §7.4 의 자연실험/대N 측정이면 effective 우월성을 검출할 수 있을 것 — 미수행.

## 부록 C: 논문 주장 시 핵심 caveat (개념적 한계)

탐색공간 수치는 견고하나, 다음은 리뷰어가 지적할 수 있어 프레이밍 주의:
- **탐색공간 ≠ 난이도 기제**: 데이터셋 난이도는 탐색공간이 아니라 **실측 모델 정확도 밴드**(easy~75%/medium~50%/hard~25%)로 경험적 보정됨(21개 파일 calibration 주석). 유형 B는 탐색공간 상수인데 난이도 증가, 유형 C는 후보집합 자체 없음 → "난이도 ∝ 탐색공간" 은 인과 아닌 사후 상관.
- **탐색공간은 제약-전(前) 상한**: minesweeper C(셀,지뢰)·sudoku 9^E·logic_grid (N!)^(K−1) 은 단서 적용 전 공간. 실제 정답은 (logic_grid 제외) 유일하므로, 이 수치는 후보 카디널리티가 아닌 **상한**. "raw upper bound" 로 명시할 것.
- **inequality (유형 A*, EN=KO 직번역)**: 탐색공간이 난이도와 **비단조**(easy 1D 큰밴드 16!≈2.1×10¹³ > medium/hard Futoshiki ≤8.1×10⁸) + tier 별 구조 전환(easy 1D→medium/hard 2D) → "공간=난이도" 로 서술 **금지**. 난이도축은 **구조 전환 + backtrack 비율(0.82→0.92)** 로 명시. (구버전 "KO 비단조/visible solver-step" 서술은 KO 재작성으로 폐기.)
- **MC task(kinship/kinship_vision)**: 탐색공간=보기 수. **kinship 은 보기 26 고정(유형 B, 난이도=노이즈 36/40/112)**, kinship_vision 은 5/10/13(유형 A). 조합적 의미 약함 → 별도 추론/노이즈 축으로 서술.
- **cipher 키후보 9**: 데이터셋이 인위 제한한 artifact → "의도적 소형 정답공간"으로 명시.
- **number_baseball (유형 A 부분)**: medium=hard 가 동일 공간 P(10,8)=1,814,400 이고 난이도 차이는 **reveal**(위치공개 0.6 vs 무공개)에서 옴 → medium→hard 를 "공간 증가"로 서술 금지.
- **cryptarithmetic (유형 A 부분, 2026-07 재작성)**: (i) 구 "10! 상수 공간" 서술은 **무효** — 3독립표 ∏P(10,kᵍ) 로 상수 아님(EN 10¹²·⁵→10¹⁴, KO 10¹⁴·⁴→10¹⁶). (ii) **EN medium≈hard(둘 다 ~10¹⁴)** 라 medium→hard 를 "공간 증가"로 서술 금지 — 차이는 공개키 가림량·추론 깊이(유효노드 2,034→10,966). (iii) **유일성이 "공개된 부분키" 조건부**임을 명시(키 없으면 비유일). (iv) 정답 = 결과 단어의 숫자값.
- **yacht_dice**: 정답이 "전체 총점"이 아니라 **SPOTCHECK_K(2/1/5) 라운드 점수 합**임을 명시(채점 단위 변경).
